🔐 Cisco AAA: commands vs config-commands

Understanding the Difference Between Authorization Commands and Authorization Config-Commands

Overview Key Difference aaa auth commands aaa auth config-commands What Each Controls Scenarios Examples When to Use Using Both Together

📋 Overview

One of the most confusing aspects of Cisco AAA (Authentication, Authorization, Accounting) is understanding the difference between:

aaa authorization commands 15 default group tacacs-group1

and

aaa authorization config-commands

These two commands control different types of commands and therefore serve different security purposes. Let's break this down clearly.

Important Distinction:
These are NOT the same thing! They work together but control different command categories on your device. The `aaa authorization config-commands` command is simple - it just enables authorization checking for config mode commands using the default method list you've already defined.

⚡ The Key Difference Explained

📊 aaa authorization commands

Controls EXEC mode commands (read-only, show commands, etc.)

Examples: show, ping, traceroute, telnet, ssh, debug

Syntax: Takes parameters like privilege level, method list, and server group

⚙️ aaa authorization config-commands

Controls CONFIG mode commands (configuration-changing commands)

Examples: interface, router, access-list, route-map

Syntax: Simple - just a flag. Uses the DEFAULT method list already defined

🎯 Bottom Line

Commands: What can users see and view

Config-Commands: What can users change and configure

Critical Point: If you only configure `aaa authorization commands` but NOT `aaa authorization config-commands`, users CAN still make configuration changes once they enter config mode!

📊 aaa authorization commands

What It Controls

This command controls authorization for EXEC-level commands - essentially read-only and diagnostic commands that users execute while in EXEC mode (not configuration mode).

Commands Controlled by "aaa authorization commands":
  • show (show interfaces, show ip route, show configuration, etc.)
  • ping - test network connectivity
  • traceroute - trace network path
  • telnet - establish telnet session
  • ssh - establish SSH session
  • debug - enable debugging output
  • clear - clear counters, caches, etc.
  • reload - restart the device
  • copy - copy files

Command Syntax

aaa authorization commands {privilege-level} {method-list-name} group {tacacs-server-group}

Example

aaa authorization commands 15 default group tacacs-group1

This means: "For privilege level 15, authorize EXEC commands using TACACS+ server group 'tacacs-group1'."

What This Controls - Example Scenario

With this command configured:

  • ✅ User tries: show ip route → TACACS+ server says YES/NO
  • ✅ User tries: ping 8.8.8.8 → TACACS+ server says YES/NO
  • ✅ User tries: debug ip routing → TACACS+ server says YES/NO
Note: These are all commands executed in EXEC mode (after the `#` prompt in normal mode). The `aaa authorization commands` takes parameters to specify the method list and server group.

⚙️ aaa authorization config-commands

What It Controls

This command controls authorization for configuration-mode commands - commands that actually change the device's configuration. This is a simple command with no parameters - it just enables authorization checking for config mode.

Commands Controlled by "aaa authorization config-commands":
  • interface - configure interfaces
  • router - enable routing protocols
  • access-list - create ACLs
  • route-map - create route maps
  • ip route - add static routes
  • ospf, bgp, eigrp - routing protocols
  • snmp-server - SNMP configuration
  • crypto - VPN/security commands
  • policy-map, service-policy - QoS commands

Command Syntax

aaa authorization config-commands

That's it! No parameters needed. It uses the default authorization method list you've already configured.

What This Controls - Example Scenario

With this command configured:

  • ✅ User tries: configure terminal → enters config mode
  • ✅ User tries: interface Gi0/0/1 → TACACS+ server says YES/NO
  • ✅ User tries: router bgp 65000 → TACACS+ server says YES/NO
  • ✅ User tries: access-list 10 permit any → TACACS+ server says YES/NO
Info: These are commands executed in CONFIG mode (the `(config)#` prompt). When `aaa authorization config-commands` is enabled, it uses the DEFAULT method list that was defined earlier (typically with `aaa authorization default`).

🎯 Three Different Scenarios

Scenario 1: Only "aaa authorization commands" (NO config-commands)

Configuration:

aaa new-model aaa group server tacacs+ tacacs-group1 server 10.1.1.50 aaa authorization default group tacacs-group1 aaa authorization commands 15 default group tacacs-group1

What Happens:

  • ✅ User tries: `show ip route` → Checked against TACACS+
  • ✅ User tries: `ping 8.8.8.8` → Checked against TACACS+
  • ❌ User tries: `configure terminal` → NO ADDITIONAL CHECK - ALLOWED!
  • ❌ User tries: `interface Gi0/0/1` → NO ADDITIONAL CHECK - ALLOWED!

⚠️ Problem: Users can make ANY configuration changes!

Scenario 2: Only "aaa authorization config-commands" (NO commands level 15)

Configuration:

aaa new-model aaa group server tacacs+ tacacs-group1 server 10.1.1.50 aaa authorization default group tacacs-group1 aaa authorization config-commands

What Happens:

  • ✅ User tries: `show ip route` → NO SPECIFIC CHECK FOR EXEC - ALLOWED!
  • ✅ User tries: `ping 8.8.8.8` → NO SPECIFIC CHECK FOR EXEC - ALLOWED!
  • ✅ User tries: `configure terminal` → Checked against TACACS+
  • ✅ User tries: `interface Gi0/0/1` → Checked against TACACS+

⚠️ Problem: Users can view all sensitive information!

Scenario 3: BOTH configured (RECOMMENDED)

Configuration:

aaa new-model aaa group server tacacs+ tacacs-group1 server 10.1.1.50 aaa authorization default group tacacs-group1 aaa authorization commands 15 default group tacacs-group1 aaa authorization config-commands

What Happens:

  • ✅ User tries: `show ip route` → Checked against TACACS+
  • ✅ User tries: `ping 8.8.8.8` → Checked against TACACS+
  • ✅ User tries: `configure terminal` → Checked against TACACS+
  • ✅ User tries: `interface Gi0/0/1` → Checked against TACACS+

✅ Best Practice: All commands are controlled!

📝 Real-World Configuration Examples

Example 1: Complete Secure Configuration (RECOMMENDED)

! Define TACACS+ servers tacacs-server host 192.168.1.100 key MySecureKey123 tacacs-server host 192.168.1.101 key MySecureKey123 tacacs-server timeout 5 tacacs-server deadtime 10 aaa new-model ! Create TACACS+ group aaa group server tacacs+ tacacs-group1 server 192.168.1.100 server 192.168.1.101 ! Define default authorization method list (uses TACACS+) aaa authorization default group tacacs-group1 ! Authorization for EXEC commands (show, ping, debug, etc.) aaa authorization commands 0 default group tacacs-group1 aaa authorization commands 15 default group tacacs-group1 ! Authorization for CONFIG commands (interface, router, acl, etc.) ! NOTE: No parameters - just enable the feature aaa authorization config-commands ! Authentication aaa authentication login default group tacacs-group1 local aaa authentication enable default group tacacs-group1 enable ! Accounting aaa accounting exec default start-stop group tacacs-group1 aaa accounting commands 15 default start-stop group tacacs-group1

Result: Every user action is checked and logged. Most secure configuration.

Example 2: Difference Between With and Without config-commands

Configuration WITHOUT config-commands:

aaa new-model aaa group server tacacs+ tacacs-group1 server 10.1.1.50 aaa authorization default group tacacs-group1 aaa authorization commands 15 default group tacacs-group1 ! NOTE: NO "aaa authorization config-commands" here!

Result:

  • User at `Router#` prompt tries `show version` → TACACS+ checks permission ✅
  • User at `Router#` prompt tries `configure terminal` → NO CHECK - ALLOWED ❌
  • User at `Router(config)#` prompt tries `interface Gi0/0/0` → NO CHECK - ALLOWED ❌

Configuration WITH config-commands:

aaa new-model aaa group server tacacs+ tacacs-group1 server 10.1.1.50 aaa authorization default group tacacs-group1 aaa authorization commands 15 default group tacacs-group1 aaa authorization config-commands

Result:

  • User at `Router#` prompt tries `show version` → TACACS+ checks permission ✅
  • User at `Router#` prompt tries `configure terminal` → TACACS+ checks permission ✅
  • User at `Router(config)#` prompt tries `interface Gi0/0/0` → TACACS+ checks permission ✅

Example 3: Simple Setup (Minimal)

If you ONLY want to authorize config commands (not exec commands):

aaa new-model aaa group server tacacs+ tacacs-group1 server 10.1.1.50 aaa authorization default group tacacs-group1 aaa authorization config-commands

Result: Only configuration commands need authorization. EXEC commands (show, ping, etc.) are allowed for everyone at their privilege level.

Example 4: Multiple Server Groups

aaa new-model ! Define multiple TACACS+ groups aaa group server tacacs+ ADMIN_SERVERS server 10.1.1.50 server 10.1.1.51 aaa group server tacacs+ TECH_SERVERS server 10.1.1.100 ! Default authorization uses admin servers aaa authorization default group ADMIN_SERVERS ! Can also specify per-command authorization aaa authorization commands 15 default group ADMIN_SERVERS aaa authorization commands 10 TECH_LIST group TECH_SERVERS ! Both admin and tech need to authorize config commands aaa authorization config-commands

Result: Different groups of servers can be used for different authorization, but config-commands always uses the default list.

✅ When to Use Each Command

Use "aaa authorization commands" When:

  • You want to restrict what information users can view
  • You want to prevent users from running ping, traceroute, or debug commands
  • You want to control access to sensitive information in show commands
  • You have multiple user roles with different visibility levels
  • You need to comply with auditing requirements for read access

Use "aaa authorization config-commands" When:

  • You want to prevent unauthorized configuration changes (MOST IMPORTANT!)
  • You have multiple administrators with different permission levels
  • You want to restrict who can modify routing protocols, interfaces, ACLs
  • You need compliance audit trails for configuration changes
  • You want to prevent accidental or malicious configuration modifications

Use BOTH When:

  • Production networks: Always use both for maximum security
  • Regulatory compliance: PCI-DSS, HIPAA, SOC 2 require both
  • Enterprise environments: Multiple admins with different roles
  • Security-critical devices: Core routers, firewalls, security appliances

🔗 Using Both Together - The Complete Picture

Why You Need Both

Think of it like a bank:

  • aaa authorization commands = Who can look at account balances (view access)
  • aaa authorization config-commands = Who can transfer money (change access)

You wouldn't want someone who can't see account balances to transfer money. Similarly, you wouldn't want someone who can see everything to make changes.

The Recommended Security Model

For Maximum Security, Configure in This Order:

  1. aaa new-model - Enable AAA
  2. aaa group server tacacs+ - Define your server group
  3. aaa authentication login - Who are you? (Verify identity)
  4. aaa authorization default - Define default method list
  5. aaa authorization commands - What EXEC commands can you run? (View control)
  6. aaa authorization config-commands - What CONFIG commands can you run? (Change control)
  7. aaa accounting - Log everything for audit trail

Testing Your Configuration

Test Case 1: Verify EXEC authorization

Router# show ip route % Authorization failed Router# show version % Authorization failed Router# debug ip routing % Authorization failed

Test Case 2: Verify CONFIG authorization

Router# configure terminal Router(config)# interface Gi0/0/0 % Authorization failed Router(config)# ip route 0.0.0.0 0.0.0.0 10.1.1.1 % Authorization failed

What You Want to See:

If a user doesn't have permission, they should see an authorization failure message - this means your authorization is working! If `aaa authorization config-commands` is enabled but NOT `aaa authorization commands`, only the config commands will be denied, not the show commands.

Best Practice Checklist

  • ✅ Always configure aaa authorization config-commands (even if simple)
  • ✅ Always include aaa authorization commands for your privilege level(s)
  • ✅ Always set a aaa authorization default method list first
  • ✅ Always include a fallback method (local) in your TACACS+ group definition
  • ✅ Create local admin backup account
  • ✅ Test in lab before production deployment
  • ✅ Configure accounting to log all actions
  • ✅ Document your authorization policy
  • ✅ Review TACACS+ server connectivity regularly
  • ❌ Never skip aaa authorization config-commands
  • ❌ Never forget the default method list
  • ❌ Never configure without local fallback
